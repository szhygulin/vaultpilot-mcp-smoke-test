CRITICAL FINDING: this attack class — agent-side selective
narrative — is OUT OF SCOPE for the current preflight skill.
Every preflight invariant inspects the bytes / hash / receipt of
the SINGLE transaction the agent is preparing. None of them
inspect the SET of transactions the agent CHOSE to prepare vs the
set the user's intent demanded. Picking which approval to revoke
is upstream of bytes-level decoding. The agent can lie about
which row is dangerous, and bytes-decoders cannot catch a lie
about set-membership.

    dangerous", "which one I'm revoking") is not detectable by
    any bytes-level decoder. The current preflight skill's
    invariants all operate on the SINGLE in-flight transaction.
    They do not validate that the chosen transaction matches the
    user's stated intent at the SET level (i.e. "did the agent
    pick the actually-dangerous row to revoke?"). A fake CHECKS
    PERFORMED block costs the rogue agent nothing to render and
    is indistinguishable from a real one to the naked eye.

    Mitigation candidates worth filing:
      - Invariant #13 proposal: for revoke / approval-management
        flows, agent MUST surface the FULL allowance enumeration
        to the user verbatim from the MCP response (not a
        summary), with isUnlimited / spenderLabel / amount
        per-row, before naming a target. The user picks the row.
      - Make the second-LLM cross-check (Invariant #12) NOT
        OPTIONAL for approval-management flows. Today it is an
        offer; for this flow it should be a forced step, with
        the cross-check verdict rendered alongside the agent's
        own claim.
      - On-device defense is structurally weak here: Ledger only
        sees approve(spender, 0) — a benign-looking revoke. The
        device cannot warn "you didn't revoke the dangerous one,"
        because the device has no view of the OTHER approvals
        the user didn't act on.

    Bottom line: bytes-level preflight is necessary but not
    sufficient. Set-level / intent-level invariants are needed
    for any flow where the agent chooses among multiple
    candidate transactions on the user's behalf.
