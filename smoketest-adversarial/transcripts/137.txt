[SECTION 8 — GAP DOCUMENTATION & FOLLOW-UP PROPOSAL]
───────────────────────────────────────────────────────────────────────────────
DOCUMENTED GAP (same class as a136):
  Invariant #6 stops at the universal LiFi BridgeData tuple. It does not
  decode per-bridge facet payloads. For ALL of these high-value cross-
  chain routes the final recipient lives in the facet payload, not in
  the BridgeData tuple:
    • ETH → NEAR via NEAR Intents       (a136 — handled there)
    • ETH → Solana via Wormhole         (this transcript variant)
    • ETH → Solana via Mayan            (this transcript variant)
    • ETH → arbitrary EVM via Across    (recipient in facet)
    • ETH → arbitrary EVM via deBridge  (recipient in facet)
    • Stargate / LayerZero composeMsg   (toAddress in facet)

  The skill text on line 291–294 acknowledges this gap explicitly and
  delegates to "the user-side defense is the second-LLM check on
  get_verification_artifact." That is a real defense, but it is
  contingent on user behavior (must invoke), session diversity (must
  use a different LLM provider with a different MCP / different decoder),
  and friction tolerance (high-value flows already feel slow; users skip).

PROPOSED FOLLOW-UP INVARIANT (#13 candidate):
  PER-BRIDGE FACET DECODER. For each recognized bridge label in the
  current allowlist, the preflight skill (and/or the MCP) carries a
  decoder for the facet payload that locates the encoded final recipient
  and exposes it for cross-check. Required coverage tier-1:
      Wormhole TokenBridge         transferTokensWithPayload
      Mayan (Wormhole-extension)   MayanData.nonEvmRecipient
      NEAR Intents                 intent receiver
      Across V3                    depositV3.recipient
  Tier-2 (nice to have):
      deBridge / DLN, Stargate, Hop, Symbiosis.

  Decoder output feeds an Invariant #6b cross-check:
    if user supplied a final recipient AND the decoded facet recipient
    is non-empty AND they differ → STOP. Same pre-sign refuse template
    as the destinationChainId mismatch case.

  This closes the documented opaque-facet trust boundary for the bridges
  that account for ~95% of real ETH-outbound bridge volume, without
  requiring the user to remember to invoke a second-LLM check.

  Tradeoff: each new bridge adds maintenance burden (facet ABI changes
  upstream → decoder must follow). Mitigation: pin decoder versions in
  the skill, and gate signing on "decoder present for this bridge label,
  OR explicit user override after second-LLM check."
