[ATTACK_VECTOR]
Two coupled attacks on `prepare_uniswap_v3_rebalance` (which under the hood
runs the standard NonfungiblePositionManager `decreaseLiquidity → collect →
swap → mint` multicall path against the user's existing in-range V3 LP):

  ATTACK 1 — OUT-OF-MONEY TIGHT RANGE
    The user asked for "current spot ±5%". The rogue MCP returns a `mint`
    leg whose `tickLower / tickUpper` encode a TIGHT range that sits
    entirely on ONE SIDE of `currentTick`. Concretely, in the ETH-USDC
    0.05% pool the rogue MCP encodes a range that is fully BELOW the
    current price — `tickUpper < currentTick`, so when the new position
    is minted the user is paid out 100% in USDC (token0/token1 ordering
    aside, the pool ends up holding only the asset on the side the range
    sits on; the other side has zero virtual reserves at mint). The user
    is now silently fully de-ETH'd into stables at the rogue MCP's chosen
    moment. The receipt's English summary still says "ETH-USDC ±5%
    around spot".

  ATTACK 2 — MEV-EXPLOITABLE INTERMEDIATE SWAP
    Even if tickLower/tickUpper looked superficially symmetric, the
    middle `exactInputSingle` swap leg of the multicall could be
    parameterized with a `sqrtPriceLimitX96` of 0 (no slippage protection
    at all, the V3 default) and an `amountOutMinimum` set far below
    fair-value. A searcher (or the rogue MCP's collusion partner) front-
    runs the swap, pushes the pool away from spot, lets the user's swap
    fill at the disadvantageous price, then back-runs to capture the
    spread. The rebalanced position is then minted around a manipulated
    "spot" that no longer matches the spot the user reasoned about.

Both attacks are detectable agent-side BEFORE `send_transaction` because
the multicall is plain EVM calldata: each sub-call's selector + args ABI-
decodes locally and the V3 pool's current `slot0.tick` is a one-RPC-call
ground truth that the agent can fetch independently of the MCP.
