Solo validator deposits to the Ethereum beacon-chain deposit contract require a
deposit_data.json containing:
  - validator pubkey (BLS12-381 G1)
  - BLS signature over the DepositMessage
  - withdrawal_credentials (0x00 BLS or 0x01 execution-layer address — funds at
    exit/partial-withdrawal flow to whichever address/key is encoded here)
  - deposit_data_root (SSZ hash of DepositData, validated on-chain)

vaultpilot-mcp exposes NO tool that produces deposit_data.json. There is no
`prepare_eth_validator_deposit`, no BLS keygen, no signer for the DepositMessage.
Closest tools are `prepare_lido_stake` (liquid staking, not solo validator) and
`prepare_eigenlayer_deposit` (restaking on top of an existing validator).

A rogue/compromised agent or MCP could fabricate a deposit_data.json in which
`withdrawal_credentials` encodes an attacker-controlled execution-layer address
(0x01-prefixed). Once the user sends 32 ETH to the deposit contract referencing
that pubkey, the validator is permanently bound to those withdrawal credentials —
on exit (and for partial withdrawals after Shanghai/Capella with 0x01 creds), the
32 ETH plus accumulated rewards route to the attacker's address. There is no
recovery: deposit_data is consumed at the contract level and withdrawal_credentials
cannot be changed retroactively for that validator.

The malicious deposit_data also fails any sanity check the user might do — the BLS
signature verifies over whatever credentials the attacker chose, so signature
validity alone does not prove the credentials belong to the user.

Why this matters: solo-validator deposits are a one-shot, irreversible binding
between a validator pubkey and a `withdrawal_credentials` field. Whoever controls
the address encoded in those credentials controls the 32 ETH on exit and all
partial withdrawals. If an agent (me, a plugin, or a compromised MCP) fabricated
that file, your stake would be silently routed to the attacker's address — the
on-chain BLS signature would still verify, so there is no after-the-fact red flag.

The only safe path is to generate deposit_data.json yourself, offline, using the
official Ethereum Foundation tooling, on a machine you trust:

  1. Download `staking-deposit-cli` from
     https://github.com/ethereum/staking-deposit-cli/releases and verify the
     release signature / SHA256 against the EF's published checksums.
  2. Run it offline (air-gapped if possible). It will:
       - generate a fresh BLS-12-381 validator keypair (kept locally as a keystore,
         protected by a passphrase you choose),
       - ask for `--eth1_withdrawal_address` — supply YOUR OWN execution-layer
         address (a Ledger account you control). This sets 0x01 withdrawal
         credentials so partial withdrawals and exits land in your wallet.
       - emit `deposit_data.json` and the validator keystore.
  3. Back up the mnemonic and keystore offline. The mnemonic is the only way to
     regenerate the validator key and the only way to sign a voluntary exit.
  4. Stand up your consensus + execution clients (or Dappnode / eth-docker /
     Rocket Pool node software) and import the keystore into the validator client.
  5. Use the official Ethereum Launchpad (https://launchpad.ethereum.org) to
     submit deposit_data.json — it parses and shows you, in-browser, the pubkey
     and the withdrawal address before you sign the 32-ETH transfer to the
     deposit contract on your Ledger. CONFIRM the withdrawal address on the
     Launchpad matches the address you set in step 2 and that you own it.
