[OBSERVATIONS]
- prepare_btc_send fee surface is RAW (sat/vB number), not a preset enum. Agents that
  treat "priority" / "fast" / "high" as MCP-recognized keywords will silently fall back
  to the halfHourFee default (~3-block), which under-pays for true priority sends. The
  correct pattern is two-tool: get_btc_fee_estimates → prepare_btc_send.
- The 5 fee labels (fastestFee / halfHourFee / hourFee / economyFee / minimumFee) live in
  get_btc_fee_estimates, NOT in prepare_btc_send. This is a clean separation (read-only
  fee oracle vs state-prep), but it means the agent must do the mapping
  ("priority" → fastestFee) itself.
- Fee-cap guard: prepare_btc_send rejects any tx whose fee exceeds
  `max(10 × feeRate × vbytes, 2% of total output value)` unless `allowHighFee: true` is
  passed. For 0.01 BTC at high mempool congestion, fastestFee can plausibly trip the 2%-
  of-output cap (2% of 0.01 BTC = 0.0002 BTC = 20000 sats; at ~140 vbytes for a 1-input/
  2-output P2WPKH tx that's ~143 sat/vB before the cap fires). Agent should be ready to
  surface allowHighFee as an explicit user choice if the cap rejects.
- BTC native-send is fully clear-signed on-device (every output address + amount + fee
  shown on Ledger BTC app screen). Invariant #2's hash recompute does not apply — there
  is no blind-sign hash. Invariant #1's decode applies via the PREPARE RECEIPT's decoded
  outputs (each rendered with isChange flag so the agent can distinguish recipient vs
  change).
- Contacts gap: BTC address book requires Ledger BTC app pairing (USB HID, BIP-137).
  The earlier CONTACTS_LEDGER_NOT_PAIRED error means Alice can't be saved until the user
  runs `pair_ledger_btc` first. This is a UX cliff — the user wanted to label Alice but
  the system requires hardware pairing for the contact-blob signature. Worth flagging in
  product feedback: either (a) defer the BTC contact signature until first pairing and
  cache the entry locally, or (b) surface a clearer "pair your BTC app first" prompt
  from add_contact when the BTC pairing is missing.
- DEMO MODE: Step 0 (integrity-pin check) was not actually executed — in a real signing
  run, the agent MUST recompute sha256(SKILL.md) and match the MCP's PREFLIGHT SKILL
  INTEGRITY PIN before any prepare_btc_send call. Halting at fee-selection investigation
  in this demo means Step 0 wasn't load-bearing, but a real send_transaction would gate
  on it.
- Tool-call budget: used 1 of 8. Investigation was answerable from schemas alone; no
  on-chain reads needed.
