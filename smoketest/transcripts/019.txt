[OBSERVATIONS]
- stETH rebasing is a UX cliff that the MCP does NOT flag automatically.
  prepare_token_send treats stETH as a generic ERC-20 (selector + decimals
  resolved on-chain), so the share-rounding artifact and the
  "balanceOf-drift revert" failure mode are invisible at the prepare layer.
  Worth flagging for v1.6: a token-class registry (rebasing / fee-on-
  transfer / blocklist-able) attached to prepare_token_send so the
  PREPARE RECEIPT can carry a "this token rebases" flag and the agent
  doesn't have to know the quirk out-of-band. Without it, every agent
  prep'ing a stETH send is one prompt-injection away from omitting the
  warning the user needs.
- wstETH recommendation is the canonical workaround. The MCP does NOT
  expose a stETH→wstETH wrap helper directly (no `prepare_lido_wrap`);
  the user would route through prepare_uniswap_swap or a 0x quote, or
  call wstETH.wrap() via a custom contract call (not currently in the
  prepare_* surface). v1.6 backlog candidate: add prepare_lido_wrap /
  prepare_lido_unwrap to round out the Lido tool family.
- Token-address spoofing is the load-bearing user-side check for any
  ERC-20 send: a fake "stETH" at a different address can serve any
  name/symbol/decimals via the standard ABI. The agent's defense is
  matching the OUTER tx.to (the token contract) against a known-good
  literal — Lido publishes 0xae7ab96520DE3A18E5e111B5EaAb095312D7fE84
  on stake.lido.fi. The vaultpilot-preflight skill doesn't ship a token
  allowlist (would be stale within months); the user-side defense is
  reading the contract address out of the CHECKS PERFORMED block.
- Invariant #11 (approval surfacing) did NOT fire here because the
  selector is transfer, not approve. Worth noting: a compromised MCP
  could swap selector to approve(spender, max_uint256) while the prose
  retelling still claims "transfer to Dave"; Invariant #1's local decode
  is what catches that — the agent never trusts the MCP's prose over
  the bytes.
- Invariant #2 anti-chain-swap is especially load-bearing for stETH:
  bridged-stETH variants exist on L2s at DIFFERENT addresses (e.g.
  stETH-on-Arbitrum is a separate contract). A chainId swap on the same
  calldata could route 0.3 units of a different token entirely. The RLP
  chainId field assertion is the defense.
- DEMO MODE: Step 0 (integrity-pin check) was not actually executed.
  In a real signing run, the agent MUST recompute sha256(SKILL.md) and
  match the MCP's PREFLIGHT SKILL INTEGRITY PIN before any
  prepare_token_send call. Halting before broadcast in this demo means
  Step 0 wasn't load-bearing for the bytes that would actually be
  signed, but a real send_transaction would gate on it.
- Tool-call budget: 1 of 8 used in this demo (ToolSearch only). Real-run
  budget projected at 5 of 8 (list_contacts, prepare_token_send,
  preview_send, get_ledger_status, plus a possible second-LLM
  get_verification_artifact if the user opts in). Comfortably under cap.
