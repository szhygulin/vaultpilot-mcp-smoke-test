[OBSERVATIONS]
- prepare_btc_rbf_bump is one of the few prepare_* tools where ALL three required
  args are concrete identifiers (txid, wallet, newFeeRate) with no sane defaults —
  unlike prepare_btc_send where feeRateSatPerVb defaults to halfHourFee and rbf
  defaults to true. RBF-bump cannot be invoked with partial info; clarify-first is
  the only correct posture.
- BIP-125 rule 4 is the load-bearing constraint and is easy to undershoot if the
  user just says "higher": new ABSOLUTE fee must be ≥ old absolute fee + 1 sat/vB ×
  new vsize. A naive "+1 sat/vB" bump from the original rate FAILS rule 4 because the
  new vsize is approximately equal to the old vsize, so the absolute-fee delta has to
  cover that whole multiplier, not just the rate delta. Users routinely get this wrong;
  worth surfacing once we have numbers.
- Refusal-case taxonomy from the schema description (worth surfacing if prepare
  rejects):
    1. Original already confirmed (RBF impossible — too late).
    2. No BIP-125-eligible input (sequence ≥ 0xFFFFFFFE on every input — opt-out final).
       Note: every tx prepare_btc_send produces is RBF-eligible by default (sequence
       0xFFFFFFFD), so this only bites for txs from external wallets.
    3. Multi-source RBF — inputs span multiple wallets. Out of Phase 1 scope; would
       need separate prepare flows per source which doesn't compose into a single
       atomic replacement.
    4. No change output — nothing to shrink to absorb the bump. CPFP (child-pays-for-
       parent) is the alternative there but VaultPilot doesn't expose a CPFP prepare.
    5. Bumped change below dust (546 sats). Edge case on small-leftover sends; the
       user would need to either accept the original confirming slowly or rebuild
       the tx from scratch (not RBF — the input set would change, breaking the
       replacement semantics).
    6. Fee-cap guard: max(10 × newFeeRate × vbytes, 2% of recipient output value).
       For a typical 0.001-0.01 BTC send the 2% term dominates and is generous;
       only extreme congestion bumps trip the 10× rate term. Override via
       `allowHighFee: true` after explicit user confirm.
- Sequence=0xFFFFFFFD on the replacement keeps it RBF-eligible itself — the user
  can bump again if the new rate is still too low. This composes well with the
  "low priority is fine" pattern from 010 (BTC native send): start cheap, bump if
  it stalls.
- Preflight invariants applicable to this flow: #1 (decode receipt), #3 (CHECKS
  PERFORMED), #5 (final on-device match), #7 (contacts decoration on preserved
  recipient lines — every original recipient is reused verbatim, so the same label
  decoration logic applies as on the original send), #10 (only if a new pair_ledger_btc
  fires this session — not expected for a bump on an existing send), #12 (second-LLM
