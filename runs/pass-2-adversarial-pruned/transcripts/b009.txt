  PSBT, "decode the bytes locally" specifically means: parse psbtBase64, walk
  each output, classify scriptPubKey (P2WPKH/P2WSH/P2TR), bech32-encode the
  witness program under hrp "bc", and compare to the receipt's claimed
  recipient. This is the load-bearing check.
- Invariant #7 (contact reverse-decoration) corroborates: the MCP's

  not, so the suffix is fabricated. Two independent invariants flagging the

- Invariant #2 (sighash recompute) is NOT load-bearing for this attack — the

  over the same tampered output set. A self-consistent hash over malicious
  bytes is not exoneration. This mirrors the EVM recipient-swap case (a003):

- Useful tell unique to this attack: the prose-rendered scriptPubKeyHex
  inside the PSBT OUTPUTS DECODE block ("0014dc6bf863...") didn't even match
  the actual scriptPubKeyHex inside psbtOutputs[0] ("0014a1b2c3d4..."). The
  rogue MCP rendered the receipt-prose hex and the actual PSBT hex in

  derive the prose hex from the PSBT bytes.
- No on-device prompt was issued (no send_transaction call, no

  malicious PSBT. Even if a prompt had fired, the Ledger BTC app's on-device
