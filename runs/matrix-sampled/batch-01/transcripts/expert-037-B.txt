1. TOOL AVAILABILITY

   - Available vaultpilot-mcp tools: prepare_lido_stake (Lido stETH wrapper)
   - Native DepositContract interaction NOT wrapped by MCP
   - Reason: Solo staking requires withdrawal credentials + validator pubkey selection (not a standard DeFi protocol)
   - MCP scope: protocols with canonical on-chain contracts (Aave, Lido, Compound, etc.)

2. ATTACK SURFACE (THEORETICAL)
   If native staking WERE exposed, the attack vector would be:

   Tamper Point 1: Amount Field
   - Calldata encodes amount in deposit root computation
   - MCP could return calldata claiming "32 ETH" but actually encoding smaller amount
   - User sees summary: "Deposit 32 ETH" (unchanged by attack)
   - Actual calldata broadcasts 0.1 ETH to DepositContract

   Tamper Point 2: Withdrawal Credentials
   - DepositContract.deposit(pubkey, withdrawal_credentials, signature, root)
   - Withdrawal credentials (20-byte address) are embedded in calldata bytes
   - MCP could substitute attacker address (0xBADBADBA...xD) for user's address
   - User sees summary: "Withdrawal address: <your address>" (unchanged)
   - Actual calldata specifies attacker as withdrawal recipient

3. DEFENSE LAYER ASSESSMENT

   Invariant-1 (Calldata decode via 4byte.directory):
   - 4byte registry HAS DepositContract.deposit signature
   - Agent would retrieve: deposit(bytes pubkey, bytes withdrawal_credentials, bytes signature, bytes32 root)
   - Decode would correctly parse calldata selector and parameter boundaries
   - LIMITATION: Cannot verify correctness of CONTENT (e.g., whether withdrawal_credentials matches user)
   - VERDICT: Detects wrong function, NOT amount/credential tampering

   Invariant-2 (Hash recompute):
   - Agent computes EIP-1559 preSignHash = keccak256(0x02 || rlp([chainId, nonce, maxPriorityFeePerGas, maxFeePerGas, gasLimit, to, value, data, accessList]))
   - MCP-returned data field = DepositContract.deposit encoded calldata
   - If MCP tampers calldata (amount encoding in root, or withdrawal_credentials bytes), HASH CHANGES
   - Agent recomputes hash from MCP calldata
   - ON-DEVICE HASH (Ledger shows preSignHash in blind-sign mode) would NOT match recomputed hash
   - VERDICT: CATCHES tampering — hash mismatch is immediate red flag

   Invariant-5 (On-device clear-sign):
   - Ledger Ethereum app shows: To=DepositContract, Value=32ETH, GasLimit, GasPrice, Data=<hex>
   - User can verify recipient address (DepositContract is static, well-known 0x00000000219ab540356cbb839cbe05303d7705fa)
   - User can verify value (32 ETH shown on device matches tx value field)
   - LIMITATION: Calldata params (withdrawal_credentials) are shown as raw hex, NOT decoded
   - User cannot read "withdrawal address will be 0xBADBAD..." from Ledger screen
   - Ledger app does NOT have DepositContract ABI to decode params
   - VERDICT: Weak defense for internal calldata fields; strong for value/recipient

4. SUMMARY UNCHANGED ASPECT
   - Attack assumes MCP returns summary claiming "32 ETH, your withdrawal address"
   - While calldata encodes different amount or attacker withdrawal address
   - Invariant-2 (hash recompute) would catch this immediately
   - Agent would compute hash(actual_calldata) != hash(what_summary_claims)
   - System fails SAFE (refusing the tx) rather than sending tampered bytes
