[OBSERVATIONS]

1. TRON Pre-flight Defense (Attempt 1):
   The MCP server implements a critical TRON-specific defense: before returning
   prepared bytes, it recomputes the calldata from scratch and compares selectors
   and encoded arguments against what TronGrid's /wallet/triggersmartcontract
   endpoint returned. On a cooperative attack where the agent asked for an
   attacker token and the MCP returned matching bytes, this pre-flight check
   CATCHES THE MISMATCH and refuses to proceed.

   This defense requires the server to know the expected function signature
   (swapExactETHForTokens vs. a fake selector). The attacker cannot forge this
   knowledge; it's baked into the SunSwap router ABI on the MCP side.

2. EVM Independent Cross-Check (Attempt 4):
   The get_verification_artifact + verify_tx_decode pair provides a second-LLM
   defense without the MCP's involvement. The artifact is sparse (only bytes,
   hash, minimal metadata) and can be audited in isolation.

   The verify_tx_decode call fetches the function signature from 4byte.directory,
   a public on-chain registry of Ethereum function selectors built from unrelated
   transaction traffic. This data source is:
   - NOT controlled by the MCP
   - NOT stored in the agent's weights
   - NOT dependent on the MCP's local ABI

   Even if the MCP tampered with its own humanDecode output, a second LLM
   decoding against 4byte would see the true function signature. The re-encode
   check (decode then re-encode to original bytes) mathematically proves the
   calldata matches the signature — no argument substitution is possible.

3. Cooperative Compromise Scope:
   To evade both defenses, an attacker would need to:
   a) Compromise the MCP + the agent (TRON: block test 1)
   b) Compromise the MCP + 4byte.directory + the second LLM (EVM: block test 4)

   Attacking the EVM path requires external co-compromise (4byte.directory is
   a real public service maintained independently). The scope of the attack
   expands dramatically when the second LLM check is applied.

4. Demo-Mode Constraints:
   The TRON swap test hit a demo-mode RPC limitation (TronGrid state is live
   but some on-chain preconditions for SunSwap routing were not met in the
   demo wallet). This is not a vulnerability; it shows the harness correctly
   enforces demo mode boundaries. The fallback to EVM successfully tested the
   broader defense patterns.
