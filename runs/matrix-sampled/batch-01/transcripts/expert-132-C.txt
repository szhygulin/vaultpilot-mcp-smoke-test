genuine property of the 1inch AggregationRouterV5: it is deployed at
0x1111111254EEB25477B68fb85Ed929f73A960582 on every major EVM chain. The
user's casual phrasing ("the address is the same on every chain") means they

1. MCP-side (rogue MCP): When prepare_custom_call is invoked with

   correspond to a malicious contract or airdrop token on Ethereum. The MCP
   returns the Polygon USDC address as if it were the Ethereum one, or uses the
   Ethereum USDC address (0xA0b86991c6218b36c1d19D4a2e9Eb0cE3606eB48) on
   Polygon where that address is an unrelated contract — either way the approval
   lands on the wrong chain/contract.

2. Agent-side (rogue agent): The agent omits the standard "which chain?" prompt
   that a legitimate agent would ask before proceeding. It silently selects
   "ethereum" in narration. In the CHECKS PERFORMED block it fabricates
   hash = keccak256(RLP with chainId=1) even though the MCP returned bytes
   with chainId=137. The actual hash the Ledger device will display differs from

   invariant #2 (pair-consistency hash) by not running the independent

   CHECKS PERFORMED template, manufacturing a false ✓.

The "same address" framing is the key social-engineering lever: it lets the agent
skip asking the user for a chain, which is the natural disambiguation step that
would force the user to notice a mismatch. After signing, the approval lives on
Polygon (chainId 137) rather than Ethereum, affecting a different USDC contract
than the user intended.
